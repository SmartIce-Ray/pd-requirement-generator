## ① 结论先行 + 要你拍

**一句话：现在这工具是「纯本地静态网页 + 你 Mac 上的品牌文件夹」，它最值钱的能力——生成完 PPT 自动归档进品牌文件夹——是个本地能力，搬到线上就不成立了。所以「上线」不是把文件丢上去那么简单，得先定清楚上线到底要解决谁的什么事，架构会因此分三种形态。**

我的推荐：**分阶段走**——先把现在的静态工具推上一个网址（半天内能完成、零成本），让别人也能用；要不要做"共享需求库"（所有人填的需求和生成的 PPT 都汇到一处、你能看能管）这个大头，等你把目标拍了再上后端。这跟你最近"先上 MVP 核心、再迭代"的打法一致。

**【要你拍 1 · 这是全案地基，先定它】上线到底为了什么？三选一（详见 ⑤ 节）：**

- **A｜只要个网址**：让研发/需求方在自己电脑上打开就能填表生成 PPT，各自下载。最快、零后端。
- **B｜要共享需求库**：所有人提交的需求 + 生成的 PPT 都进一个中心库，你能统一看、统一管、按品牌归档。要后端 + 存储，是真正的"前后端"工程。
- **C｜B 之上再加 AI 辅助**：在"需求描述/参考图点评"旁边加个按钮，AI 帮研发方理解审美意图、补全描述（走 OpenRouter）。

> 概念先说：**静态网页**=只有 HTML/CSS/JS、不需要服务器跑逻辑的页面，往任何静态托管平台一放就能访问。**后端**=一台一直在跑的服务，负责存数据、做鉴权、对外提供接口（这工具现在一行后端都没有）。

## ② 三句话背景（现状→变化→所以）

- **现状**：这工具是你 2026-06-22 做的本地小工具——双击 `启动.command` 在本机 `localhost` 起个服务，浏览器里选品牌、传参考图、填四段需求、一键生成 PPT，然后**自动写进 `品牌/需求/<需求名>/`**。归档靠的是浏览器的 File System Access API（下面解释），只在本机有效。[G]
- **变化**：你要"正式部署上线"，意味着访问入口从"你 Mac 上的 localhost"变成"一个公网网址，别人也能开"。
- **所以**：访问者从"只有你"变成"多人/多台电脑"，而那台决定一切的本地能力（自动归档进品牌文件夹）对别人的电脑不成立——这一步把工具从"你的本地归档器"逼成了一个要重新想清楚数据落在哪的多人产品。这就是为什么要先拍方向，而不是直接 deploy。

## ③ 现状盘点 · 凭什么这么说（每条：数据行 + 推断行）

每条第一行是我核到的事实，第二行是由它得出的判断。

**1｜前端：纯静态，可直接托管，零改动**

- 数据[G]：`app/` 全是静态文件（`index.html` + `js/*.js` + `styles.css` + `vendor/pptxgen.bundle.js` 452KB 离线 PPT 库）；代码内**无任何** `localhost`/写死的 URL/接口调用，全是相对路径；没有构建步骤、没有运行时依赖。
- 推断[H→G]：可以原封不动丢到任何静态托管（Cloudflare Pages 等），前端不用改一行就能打开。这一半是稳的。

**2｜后端：不存在**

- 数据[G]：全项目零接口调用、零服务端代码、零数据库。`_serve.py` 只是本地开发用的小静态服务器（带禁缓存头），不是后端，也不该上线。
- 推断[G]：任何"多人共享/集中存储/AI"诉求都得**从零起一个后端**，不是改造现有的。

**3｜Cloudflare：装了工具、没登录**

- 数据[G]：`wrangler`（Cloudflare 的命令行工具）已装 v4.84.0，但 `wrangler whoami` 没返回身份 → **未登录**。
- 推断[G]：要用 Cloudflare 得先登录授权一次。

**4｜Zeabur：命令行没装、上面没东西**

- 数据[G]：`zeabur` CLI 未安装；既然项目没 git、没后端，Zeabur 上自然也没有任何对应服务。
- 推断[H]：Zeabur 这条线只有在选 B/C（要后端）时才需要启用；选 A 用不到。

**5｜GitHub：工具在、登录是坏的、且还没有这个仓库**

- 数据[G]：`gh` v2.90.0 已装，但登录报错——`Failed to log in to github.com account YukikoYoung (keyring)`，意思是它记得有个账号 `YukikoYoung`，但读不出存在系统钥匙串里的令牌。这个项目也**还不是 git 仓库**、GitHub 上没有对应 repo。
- 推断[H]：上线前要先把 GitHub 登录修好（重新 `gh auth login` 或修钥匙串），再建仓库。**【要你拍 2 见 ⑦ 节：用哪个 GitHub 账号】**

**6｜git 本体：项目没纳入版本管理、全局身份是占位符**

- 数据[G]：`git rev-parse` 显示这不是 git 仓库；`git config --global user.name/email` 是默认占位符 `Your Name / your.email@example.com`。
- 推断[G]：要先 `git init` + 配好真实提交身份，否则提交记录会挂错名字。

**7｜命门：自动归档是本地能力，搬上线就断（最重要的一条）**

- 数据[G]：归档靠浏览器 **File System Access API**（让网页在你授权后直接读写你电脑上某个文件夹的能力）。它要求页面跑在"安全环境"——`https://` 或 `localhost` 才行。
- 推断[G+H]：搬到 `https://` 的线上网址后，这个 API **技术上还能用**，但它写的是**访问者自己电脑上**的文件夹。只有你（和把整个项目 clone 下来的人）电脑上才有那套 `产品研发需求/品牌/...` 文件夹；研发/需求方在自己电脑上打开，**没有这套文件夹**，只能退化成"下载到下载文件夹"。也就是说——**"自动归档到一处"这个核心价值，一旦多人用就散了**。要保住它，只有上后端（B）。这条若你原本以为"上线后还能自动归档进我的 Mac"，那是个误解，得纠正。

## ④ 术语随读随懂（本节出现的，先概念后名词）

- **File System Access API**：浏览器让网页"经你点头后直接读写你电脑某文件夹"的能力。只有 Chrome/Edge 支持，Safari/Firefox 不支持（会降级成普通下载）。这工具的"自动归档"就靠它。
- **静态托管 / Cloudflare Pages**：把一堆静态文件（HTML/JS/CSS）放上去、自动给你一个带 https 的网址、全球加速、免费额度足够。Pages 是 Cloudflare 的静态托管产品；它也能接 GitHub，代码一推自动更新网站。
- **后端 + 对象存储 + 数据库**：后端=一直在跑的服务；对象存储（如 Cloudflare R2）=专门存文件（比如生成的 PPT）的地方；数据库（如 Postgres）=存结构化记录（谁、什么品牌、什么需求、什么时候提交）的地方。B 方案这三样要一起上。
- **Zeabur**：一个国内可用、push 代码自动部署后端服务的平台，适合放 Node/NestJS 后端（你别的项目就用它）。
