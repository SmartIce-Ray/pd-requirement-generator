# 选品灵感库 × 研发需求一体化 — 设计稿

日期：2026-06-25 · 状态：已批准（基于 ~/.claude/plans/witty-marinating-wave.md）

## 背景与目标

把一条完整工作流做进一个工具：**刷到产品图 → 存图打标（采集）→ 总览翻阅、勾选（选品）→ 一键生成研发需求 PPT（立项交研发）**。

研发需求 PPT 生成器已存在并上线（本项目，CF Pages，pd.smartice.ai）。决策：**就地扩展本项目**，加“选品库”+ 跨设备后端，合并成一个项目/一套部署，域名沿用 pd.smartice.ai。不新建、不迁 Workers。

## 现状（底座）

- CF **Pages** 纯静态前端（`app/`），`wrangler.jsonc` 锁 `pages_build_output_dir: app`，无后端。
- 可复用模块：`deck.js`（pptxgenjs 出 PPT，单产品）、`config.schema.js`（品类5+味型库+按品类子字段）、`config.brands.js`（4品牌）、`app.js`（单产品表单编排）、`images.js`（图片→长边1600 JPEG dataURL，HEIC 拦截/EXIF 转正）、`store.js`（localStorage 草稿）、`fs.js`（File System Access 落地/下载）。
- pptxgenjs 本地 vendored：`app/vendor/pptxgen.bundle.js`。
- 缺口：无持久化后端 → 选品库跨设备存图必须新增后端。

## 锁定决策

1. 选品**每图一条**；品牌**可多选**（4品牌）；分类**预设(schema 5类)+可自定义**。
2. 总览按品牌/分类筛 +「未整理」筛；单人**密码门**。
3. 研发需求 = **多产品合成一份清单**（勾选 N 产品→各占一段→一份 PPT）。
4. **生成即走**（localStorage 草稿，服务器不留研发需求历史）。
5. 存储 = **跨设备后端**（图存云端）。
6. 就地扩展 CF Pages 项目，加 Pages Functions + R2 + D1，域名沿用 pd.smartice.ai。

## 架构

CF Pages 不变，新增 **Pages Functions（后端 API）+ R2（存图）+ D1（选品元数据）**；前端 `app/` 加选品库界面与“选品库 | 研发需求”切换。绑定经 `wrangler.jsonc` 的 `d1_databases`/`r2_buckets`，Functions 用 `context.env.DB` / `context.env.IMAGES`；后端绑定行为以 `wrangler pages dev` 本地实测 + preview 部署为准。

### 数据模型（D1）
```sql
CREATE TABLE inspirations (
  id TEXT PRIMARY KEY, image_key TEXT NOT NULL, image_type TEXT,
  brands TEXT NOT NULL DEFAULT '[]', category TEXT,   -- NULL=未整理
  notes TEXT NOT NULL DEFAULT '', created_at INTEGER NOT NULL );
CREATE INDEX idx_insp_created ON inspirations(created_at);
CREATE TABLE categories ( id TEXT PRIMARY KEY, name TEXT NOT NULL UNIQUE, sort_order INTEGER NOT NULL );
-- seed: 菜品/小吃/饮品/甜品/主食
```
品牌为常量（复用 `config.brands.js`）。R2 桶 `pd-inspirations` 存 `images.js` 处理后的 JPEG，key=记录 id。

### API（Pages Functions，`/api/*` 与 `/api/img/*` 走密码门）
- `POST /api/login`、`GET /api/config`（品牌+分类）
- `GET /api/inspirations?brand=&category=&untagged=1`、`POST`（multipart 单图→R2 put + D1）、`PATCH /api/inspirations/:id`、`DELETE /api/inspirations/:id`（D1+R2）
- `POST /api/categories`、`DELETE /api/categories/:id`
- `GET /api/img/:key`（鉴权后取 R2）
- 鉴权：`Authorization: Bearer == env.APP_PASSWORD`（v1 密码即令牌）。

### 前端
- **选品库**（`library.js`+`apiclient.js`）：登录→上传（多选图，`images.js` 处理→逐图 POST，每图品牌(多选)/分类/想法，批量套用，可留空）→ 总览（品牌/分类/未整理筛 + 卡片网格，缩略图 `/api/img/:key`）→ 详情（改标签 PATCH/删）。想法 textContent 渲染防 XSS。
- **勾选→研发需求**（`reqform-multi.js`，生成即走）：选品模式勾选 N→N 个产品块（复用 `app.js` 渲染函数），预填品牌(首个)/品类(驱动子字段)/参考图(R2 取→dataURL)/想法→描述；补 需求名/呈现/规避/味型/子字段/承接/排期，可加图；草稿 localStorage；生成→`buildDeckMulti`→下载或 `fs.js` 落地。

### deck.js 多产品扩展
抽页面函数 `addCover/addDescPage/addRefPages/addRequirePage/addSchedulePage`（复用 `head/foots/newContent/card/fitH/fitBox`）；保留 `buildDeck(data)`（单产品/单测）；新增 `buildDeckMulti(products, meta)` = 清单封面 + 目录页 + 每产品段（段头“产品i·品牌·需求名”+描述/参考图/要求）+ 末尾共享落地节点页；页脚“产品研发需求清单·日期”+全局页码。

## 安全
- 密码门 secret `APP_PASSWORD`（`wrangler pages secret put`，绝不入仓库）。
- 图片私有：`/api/img/:key` 鉴权后从 R2 取，不公开桶。
- 用户文本一律 textContent / 属性赋值，不 innerHTML 注入。

## 验证标准
- 自动：`npm test`——`buildDeckMulti` Node 单测（N 产品→页数正确、合法 pptx、不溢出）+ 后端纯逻辑单测（鉴权/校验/brands 筛选/id）。
- 本地：`wrangler pages dev` 全流程；移动 430 + PC 1440 双端（Playwright）。
- 部署：`deploy:preview` 实测后端绑定 + 真机手机上传 + 多产品 PPT 打开核对 → `deploy` 生产 pd.smartice.ai 冒烟。

## 不做（YAGNI）
账号系统、研发需求服务器留痕/排期状态、选品“已立项”标记、AI 自动打标、缩略图、导出。
